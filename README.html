3Secure Boot and Secure Signing overview

Â 

Protection of system firmware against malicious attack is paramount to server security. If an attacker is able to inject malicious code at the firmware level, no security measure at the operating system level can fully guarantee the trust of the system. IBMÂ® POWER servers support secure boot of system firmware to ensure the system boots only authorized firmware. When the system boots, each firmware component is verified against a cryptographic signature and integrity-checked against a secure hash of the component. If any check fails, secure boot prevents the system from booting until the problem is corrected.

3.1Secure boot protection of platform firmware

When a POWER server boots, the boot process loads a series of executable firmware components from flash memory. Â When secure boot is enabled, each component in the boot sequence will verify the integrity and the authenticity of the next before allowing that component to run. Â The integrity is verified by means a secure cryptographic hash, such as SHA512, and the authenticity is verified by a cryptographic signature, such as ECDSA (elliptic curve digital signature algorithm).

Secure boot works by establishing a chain of trust for all executable components. Â This chain of trust is anchored in hardware, meaning that secure boot trusts the very first bit of code to execute because it is stored immutably in the machine hardware. Â Each component that executes subsequently, however, is loaded from an unprotected location in flash memory and so is untrusted until verified. Â Secure boot stores each of these components in a secure container that includes a cryptographic signature. Â For every component loaded, secure boot will authenticate the component against this signature, and stop the boot if the check fails.

3.2Secure boot security domains

The POWER secure boot function comprises two security domains, the firmware domain and the operating system (OS) domain. Â The firmware domain governs the boot of platform firmware up to and including Petitboot (the firmware-level bootloader). Â The operating system domain governs the boot of the target OS or hypervisor. Â This is shown in figure 1.

The keys are asymmetric keys, meaning that each has a public portion and a private portion. Â The private portion, or private key, must be protected (kept secret) by the key owner, stored ideally in a hardware security module (HSM). Â The public portion, or public key, may be distributed openly. Â The private keys are typically controlled by the platform vendor: IBM or the ODM (original design manufacturer). Â Or, the private keys may be controlled by the end-customer (platform owner). Â 

The private keys are never exposed to the secure boot process and are not added to the firmware container. Â The public keys are added to the firmware container so that they are available for the signature check during boot.

To anchor the key hierarchy in hardware, a SHA512 hash of the public portion of the three HW Root keys is installed in protected flash memory in the POWER processor SEEPROM. Â The process for installing this hash is referred as the imprinting process, and this is done (normally once) by the platform vendor or platform owner. Â When secure boot loads containers from PNOR, it checks the hash of the HW keys in each secure container against the value stored in SEEPROM. Â This is how secure boot verifies that the keys used to sign the firmware are properly authorized.

The current OpenPOWER secure boot implementation specifies the use of 512-bit ECDSA for all keys.

3.4POWER secure boot flow

Secure boot establishes a chain of trust for all executable components by requiring each component to verify the next. Â We say the chain of trust is anchored in hardware, because the first bits of code to execute are stored immutably, either in ROM, or in protected area of NVRAM that cannot be updated without the authority of the HW Root keys or by asserting physical presence.

The remaining firmware is stored in PNOR, which is an unprotected region of flash memory, and secure boot considers the PNOR to be untrusted. Â To protect the firmware in PNOR, each component us stored in a secure container that includes a cryptographic signature. Â As each module is loaded, the secure boot code in the previous module will use this signature to verify the integrity and authenticity of the module to be executed. Â If any check fails, secure boot halts the system.

3.5Secure firmware containers

Each component of POWER firmware is packaged in a secure container that holds the hash and signatures required to support secure boot verification. Â The container is in fact a metadata header prefixed to the firmware image.

Figure 2 shows a simplified view of the container layout. Â The header is created during the firmware build process, or it may be created at some later packaging step. Â To build the header, the process performs the following steps:

  1. 1.Adds a (SHA512) hash of the firmware image (the container payload) to a region of the header known as the firmware header. Â Â 

  2. 2.Requests a signature over the firmware header to be created by the holder(s) of the three FW keys, then adds these signatures to an adjacent region immediately following the firmware header.Â 

  3. 3.Calculates a hash of the three FW keys and adds this to a region of the header known as the prefix header.Â 

  4. 4.Requests a signature over the prefix header to be created by the holder(s) of the three HW keys and adds these signatures to a region immediately following the prefix header. Â (In a manner of speaking, the FW keys are the â€œpayloadâ€ signed by the HW keys.) Â Â 

  5. 5.Adds a copy of the public HW keys to a region called the hardware header, and a copy of the FW public keys to the â€œpayloadâ€ region of the prefix header.Â

3.6Key management and Separation of Duties

The private portions of the HW and FW keys will be managed by IBM, the ODM vendor or the end-customer, depending on the procurement method and the application. Â An OpenPOWER server purchased from IBM will ship with a hash of the IBM HW keys installed in SEEPROM. Â Customers will obtain their firmware updates from IBM, through IBM Fix Central, for example. Â When IBM packages the firmware for release they will build the containers using IBM-managed keys. Â IBM's manufacturing process specifies that these keys are stored in an HSM, where the private keys are never actually exposed outside the HSM. Â When the server is in secure mode, it will refuse to boot any firmware not signed by IBM.

Similarly, an OpenPOWER server purchased from an ODM vendor will ship with a hash of the ODM's HW keys installed in SEEPROM. Â Customers will obtain their firmware updates from the ODM, through their supported channels. Â 

As mentioned, the use of (up to) three keys in each set, HW (A,B,C) and FW (P,Q,R), permits a separation of duties for each signing operation. Â When all three are used, secure boot requires a signature by all three keys to be present in the container header. Â These means that all three FW key holders must â€œsign offâ€ on a firmware module to make it bootable, and all three HW key holders must sign off on the FW keys to authorize their respective modules.

4.4Signtool modes of operation

Signtool supports three modes of operation, or signing modes:

  • â€¢Local mode (a.k.a. development mode) â€“ Build the container and sign using locally available private keys. Â Signatures are generated using simple openssl operations. Â Because the private keys are exposed on the local system (the build machine), this mode should be used only for development signing, or when the user is confident that the build machine is secure against unauthorized access.Â 

  • â€¢Independent mode â€“ Generate the signing requests locally and export the requests for signing externally.   • â€¢Production mode â€“ Build the container locally and interface with the remote signframework to retrieve signatures and (public) keys as needed. Â In other words, if the HW key signers have authorized a set of n FW keys P,Q,R, where n = 1-3, signatures by n FW keys must be present in the container. Â There is no such restriction between groups of FW signers, however. Â If the HW key signers authorize more than one group, one group may employ keys P,Q,R, while another group employs only P, and so on.

9Using VALIDATE and VERIFY

Signtool includes a built-in container validation and verification function that may be used when signtool stand-alone or under op-build. Â The operation is controlled by three configuration properties, SB_VALIDATE, SB_VERIFY and SB_PASS_ON_ERROR. Â These properties may be set by environment, command-line or INI.

The VALIDATE function emulates all intra-container checks normally done during secure-boot. Â This includes:

  • â€¢Validate all HW key signatures in the container, using the public keys from the container header.Â

12Threats and vulnerabilities Â

The POWER firmware signing process aims to produce an installable firmware package that that can be trusted as authentic, regardless of the method by which the package was obtained (by download, from install media, etc.). Â The POWER firmware Secure Boot function can provide this assurance by checking cryptographic signatures contained in the firmware package (the component container metadata), at the point the firmware is loaded, and comparing the signing keys to a known-good value â€œburnedâ€ into to system NVRAM. In this analogy:

  • â€¢HW key signer is the Authority (analogous to the CA)Â 

  • â€¢FW key signer is the Requester (analogous to the entity making the request to the CA)Â

The public keys in the archive are analogous to the CA and Requester public keys.

An attack on the "hand-off archive" seems no more useful than an attack on a CSR. Â What could the attacker hope to gain? Â Even if the attacker was able to replace the legitimate public keys and X509 in the CSR, and fool the Requester into using them, the Requester's transactions would be exposed as fraudulent (or at least, unverifiable) by the first third-party that checked them against the CA. 12.2.1How signer authorization works with signframework

The signframework client provides an interface to the signing server. Â The signing server is the machine physically hosting the Hardware Security Module (HSM, a.k.a. crypto card).

To retrieve the necessary artifacts (keys and signatures) to build the container, the signtool makes a request to the signframework via the signframework client. Â The signframework, in turn, passes the request to the signing server, and then passes the returned artifacts back to signtool. Â Only public keys are returned by the signing server; the private keys are generated and stored within the HSM and are never exposed. Â The requested signatures are generated on the signing server by the HSM.

The signframework employs a "SFTP dropbox" method of message passing, which the signframework client may use to make requests to the signing server. Â The signing server finds the requests in the dropbox, processes them, and returns the responses back to the dropbox where the client can pick them up. Â This provides a simple message queuing implementation, although there is no "publish/subscribe" mechanism: both the client and the server must poll the dropbox to find requests and responses.

To provide authorization to the signframework, the client uses a standard, asymmetric SSH key. Â The public part is registered at the signing server, to allow the client to access the dropbox. Â The private part is held by an individual user (the "signer"), who initiates the connection to the SFTP dropbox. Â To provide a higher level of security, this private SSH key must be password-encyrpted as stored on disk. Â The signframework client enforces this restriction, stubbornly refusing to handle any key that is not in encrypted format. Â To use the key, the signer enters the password upon every request to the dropbox. Â The signframework client prompts for the password, accepting the input via (echo-suppressed) keyboard entry, reading the password in plaintext. Â The plaintext password is held only in the signframework client process memory; it is never written to disk [are we sure of this? 12.2.2Exploits and vulnerabilities of the signframework client

The system where the signframework client runs becomes a significant part of the overall attack surface. Â If an attacker is able to compromise this system, he may be able to fool the signer into signing some unauthorized payload, or worse: fool the signer into entering the password so it may be captured in plaintext. Â If the attacker can get the password, and the encrypted SSH key file (which is likely stored on disk, on or accessible by, the signframework machine), he may then be able to make requests to the signing server to sign some unauthorized payload with the signer's credential. Â This would then allow the attacker to introduce the unauthorized payload to a POWER server running in secure mode, and start the execution of some code that would otherwise not be permitted.

The most direct way for an attacker to achieve this is hack the signtool code, to sign some arbitrary payload (or hash thereof); or to hack the signframework code, to capture the signer's password in plaintext. Â This may be mitigated by instructing (and/or enforcing) the signer to always pull the signtool and signframework code directly from a trusted source (e.g. the upstream repository) and rebuilt it, immediately prior to the signing operation; OR to otherwise check the integrity of the code, for example, by comparing the (hash of) the executable files to a "reference manifest" of trusted code. Â Or, the signtool and signframework packages could be themselves packaged in some secure container (i.e. "code signed") to better ensure their integrity.

But the vulnerably runs deeper that this, and touches the very essence of secure computing: If the machine on which the signtool or signframework code runs cannot be trusted, then "all bets are off" with regard to code integrity. Â If an attacker can gain control of the machine (or at least, the singer's login to the machine), the attacker could alter the environment to make it *appear* that the above integrity checks were successful, when in fact it is the attacker's code executing (or likely, circumventing) these checks. Â Simply put, if the signer cannot trust the machine where the signframework client runs, there is no way for the signer to be sure that the correct artifacts were retrieved, a legitimate payload was signed, or that his/her SSH key password is not exposed.
